# 需要手工执行的操作

## 优先级排序

### 🟢 P0：部署后必须验证（立即执行）

#### 1. 验证环境变量配置
**操作**：检查部署环境的环境变量
**位置**：部署平台（Vercel、Railway、自建服务器）的环境变量设置
**检查项**：
- `DUAL_AI_ENABLED=true`
- `METASO_API_KEY` 有效且未过期
- `METASO_API_BASE_URL` 正确（默认 `https://api.metaso.cn`）
- `AI_TIMEOUT_MS`、`AI_RETRY_MAX` 等配置合理

**验证方法**：
- 登录部署平台控制台
- 查看环境变量列表
- 确保与 `.env.local.example` 一致

**预期结果**：所有变量已设置且值正确
**异常判断**：如 `METASO_API_KEY` 缺失或错误，深度诊断将 fallback 到 DeepSeek

#### 2. 验证日志系统
**操作**：确认结构化日志能被收集和查询
**位置**：日志聚合系统（Vercel Log Drains、CloudWatch、ELK 等）

**检查项**：
- 日志标签：`DiagnoseAPI`、`AIRouter`、`MetasoProvider`、`DeepWorkflow`
- 日志级别：INFO、WARN、ERROR
- 日志格式：JSON 结构化

**验证方法**：
1. 发送一次 deep 模式诊断请求
2. 在日志系统中搜索上述标签
3. 确认能看到完整的调用链日志

**预期结果**：
```
[INFO] DiagnoseAPI - 诊断请求参数检查 {...}
[INFO] AIRouter - 任务路由开始 {...}
[INFO] MetasoProvider - 断路器状态检查 {...}
```

**异常判断**：如看不到日志，检查日志配置或权限

#### 3. 验证 Fallback 行为
**操作**：模拟 Metaso 故障，确认 fallback 正常工作
**位置**：生产或预发布环境

**步骤**：
1. 临时设置错误 `METASO_API_KEY`
2. 发送 deep 模式请求
3. 观察日志和响应

**预期结果**：
- 日志中出现 `Falling back to deepseek`
- 响应中 `metadata.deep_fallback_reason` 为 `research_memo_failed`
- 请求成功完成（使用 DeepSeek）

**异常判断**：
- 如请求失败：检查 AI router 逻辑
- 如无 fallback 标记：检查 workflow.ts 修改

### 🟡 P1：功能验收（部署后 24 小时内）

#### 4. 验证深度诊断质量
**操作**：对比 deep 模式与 basic 模式的诊断结果
**位置**：生产环境用户界面

**步骤**：
1. 使用同一份简历和 JD
2. 分别请求 basic 和 deep 诊断
3. 对比报告差异

**预期结果**：
- deep 报告应包含更多行业洞察（如 Metaso 正常工作）
- 或明确标记 fallback（如 Metaso 失败）
- 响应时间合理（deep 模式可能稍长）

**异常判断**：
- deep 与 basic 结果完全一致：可能 Metaso 未工作
- deep 响应时间异常长：检查超时配置

#### 5. 验证断路器恢复
**操作**：观察断路器自动恢复机制
**位置**：生产环境日志

**步骤**：
1. 连续触发多次 Metaso 失败（如 6 次超过阈值 5）
2. 观察断路器打开日志
3. 等待 60 秒（默认 `AI_CIRCUIT_OPEN_MS`）
4. 再次请求，观察断路器是否自动关闭

**预期结果**：
- 失败超过阈值后出现 `断路器打开` 日志
- 60 秒后出现 `断路器自动关闭` 日志
- 后续请求重新尝试 Metaso

**异常判断**：
- 断路器未打开：检查阈值配置
- 断路器未自动关闭：检查时间计算

### 🔵 P2：监控与优化（部署后一周内）

#### 6. 监控日志体积
**操作**：评估新增日志对存储的影响
**位置**：日志存储系统

**检查项**：
- 每日日志增量
- 单个请求的日志行数
- 结构化日志字段大小

**行动**：
- 如日志量过大，考虑调整日志级别
- 或优化日志字段（减少冗余）

#### 7. 验证生产流量
**操作**：观察真实用户使用 deep 模式的情况
**位置**：业务监控面板

**指标**：
- deep 模式请求量
- fallback 发生率
- 平均响应时间
- 错误率

**行动**：
- 如 fallback 率过高：调查 Metaso 服务稳定性
- 如响应时间超标：调整超时配置

## 操作风险说明

### 低风险操作
- 环境变量检查：只读操作，无影响
- 日志验证：只读操作，无影响
- Fallback 测试：临时错误配置，可立即恢复

### 中风险操作
- 断路器测试：可能短暂影响 Metaso 调用，但 fallback 机制保证服务可用
- 生产流量监控：只读，但需注意监控系统权限

### 注意事项
1. **API Key 安全**：测试时不要将真实 API key 提交到代码仓库
2. **测试数据**：使用模拟简历数据，避免真实用户数据
3. **服务影响**：避免在高峰时段进行断路器测试
4. **回滚准备**：保留修复前代码快照，必要时可快速回滚

## 紧急情况处理

### 场景 1：日志系统过载
**现象**：日志量激增，存储告警
**应急措施**：
1. 临时调整日志级别为 WARN
2. 修改 `lib/error-handler.ts` 中的日志配置
3. 部署热修复

### 场景 2：Fallback 失效
**现象**：Metaso 失败导致请求完全失败
**应急措施**：
1. 检查 `lib/ai/router.ts` 中 fallback 条件
2. 验证 `AIProviderError.isRetryable` 逻辑
3. 临时禁用 Metaso：设置 `DUAL_AI_ENABLED=false`

### 场景 3：断路器误锁
**现象**：断路器打开后未自动恢复
**应急措施**：
1. 检查系统时间同步
2. 重启服务进程（重置内存中断路器状态）
3. 调整 `AI_CIRCUIT_OPEN_MS` 为更短时间
